Version control

Version control is a key part of our development workflow. Everything we do centers around version control to some degree.

Git is our version control engine of choice. We use GitHub for our Git hosting.

Following are the rules we've defined around Git and our workflow.

  • The master branch must always be complete, working and finished. We always deploy master branch to production.
  • All development must take place on branches. Branch structure should be master > feature/epic > user story.
  • All code must be peer-reviewed through a GitHub pull request.
  • A development branch can only be merged into master or a feature/epic branch if you can do so without conflicts (we use GitHub to visualise this).
  • Git commits should be smaller, and task focused. More commits is better than fewer.
  • When merging a user story branch back into a feature/epic branch, always reference the user story ID.
